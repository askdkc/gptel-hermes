#+title: gptel-hermes
#+author: dkc

* 概要

=gptel-hermes= は、Nous Researchが開発したHermes Agentの機能の一部を、Emacsのgptel経由のチャットで利用可能にする独自のEmacs Lispパッケージです。

#+begin_example
Emacs / gptel
  └── gptel-hermes.el
       ├── skills/**/SKILL.md の索引
       ├── system prompt への索引・メモリ注入
       ├── hermes_skill_view tool
       ├── hermes_skill_validate tool
       ├── hermes_skill_create tool
       ├── hermes_memory tool
       ├── hermes_org_agenda tool
       ├── hermes_org_task tool
       ├── workspace/file/Git/terminal runtime tools
       └── bounded/explicit Elisp tools
#+end_example

Hermes Agentの機能をEmacsで利用するため、gptel向けに独自実装しています。

runtimeはEmacs-native実装です。workspaceはgptelバッファ単位で一度だけ捕捉され、ホーム配下のprojectだけ自動選択されます。未設定の場合は =gptel-hermes-set-workspace= で明示します。TRAMPは対象外です。ファイル、patch、skill更新はworkspace・サイズ・secret path・SHA-256を検証します。terminalはargv配列で実行し、shell文字列を組み立てませんが、OS sandboxではありません。

* セットアップ

** use-package と :vc でインストール（Emacs 30以降）

Emacs 30以降では、組み込みの =use-package= と =:vc= を使ってGitHubからインストールできます。初回評価時にリポジトリが取得され、 =gptel-hermes= がロードされます。

#+begin_src emacs-lisp
(use-package gptel-hermes
  :vc (:url "https://github.com/askdkc/gptel-hermes.git"
       :rev :newest)
  :custom
  (gptel-hermes-skills-directory
   (expand-file-name "~/.gptel-hermes/profiles/main/skills"))
  (gptel-hermes-home
   (expand-file-name "~/.gptel-hermes"))
  :config
  (gptel-hermes-global-send-mode 1)
  (add-hook 'gptel-mode-hook #'gptel-hermes-enable))
#+end_src

この方法にはGitが必要です。 =gptel-hermes-skills-directory= と =gptel-hermes-home= は既存のHermes Agentの環境と分離した上記のディレクトリを初期設定として推奨します。設定しない場合も同じディレクトリがデフォルトで使われます。

** 手動でインストール

#+begin_src emacs-lisp
(add-to-list 'load-path "~/YOUR-CODE-PATH/gptel-hermes")
(require 'gptel-hermes)
(gptel-hermes-global-send-mode 1)
(add-hook 'gptel-mode-hook #'gptel-hermes-enable)
#+end_src

gptel バッファで手動有効化する場合:

#+begin_src emacs-lisp
M-x gptel-hermes-enable
#+end_src

* 使い方

** 最初の起動

1. gptelでチャットバッファを作る。
2. =gptel-hermes-enable= を実行する（通常は =gptel-mode-hook= から自動実行）。
3. workspaceが未設定なら =gptel-hermes-workspace-status= で確認し、必要なら =gptel-hermes-set-workspace= でディレクトリを選ぶ。
4. =gptel-hermes-prompt-inspect= で、現在のバッファに実際に設定されたsystem promptを確認する。

#+begin_example
;; gptelバッファで一度だけ明示する場合
M-x gptel-hermes-set-workspace

;; 現在のworkspaceを確認する場合
M-x gptel-hermes-workspace-status
#+end_example

workspaceはバッファごとに初回有効化時の =default-directory= から捕捉されます。ホーム配下のprojectは自動選択されますが、ホームそのもの、ホーム外のディレクトリ、TRAMPは自動選択されません。自動選択されなかった場合は、推測で別の場所を使わずエラーになります。

** チャット中の使い方

通常はユーザーがtool名を入力する必要はありません。モデルに次のような作業を依頼すると、gptel-hermesが適切なtoolを呼びます。

#+begin_example
このworkspaceのsrc/example.elを読み、関連するskillがあれば先に読み込んでください。
変更案をunified Git patchで提示し、適用前にcheck-onlyで検証してください。
#+end_example

必要なskill本文やresourceを直接指定して確認する場合:

#+begin_example
hermes_skill_view(name="software-development/example")
hermes_skill_view(name="software-development/example", resource="references/example.txt")
hermes_skill_resource_path(name="creative/comfyui", resource="scripts/run_workflow.py")
#+end_example

Orgタスクを扱う場合は、先に =hermes_org_agenda= で現在のfile・line・heading・keywordを取得し、その値を =hermes_org_task= に渡します。一般のfile/terminal toolでOrgファイルを直接書き換えないでください。

** グローバル送信

=gptel-hermes-global-send-mode= を有効にすると、すべてのbufferで =C-c RET= が
=gptel-hermes-send= になります。現在のbufferでHermesが未有効なら最初に
=gptel-hermes-enable= を実行し、その後、同じbufferから =gptel-send= します。

#+begin_src emacs-lisp
(gptel-hermes-global-send-mode 1)
#+end_src

送信範囲はgptel標準と同じです。active regionがあればregionだけを送り、なければ
buffer先頭からpointまでを送ります。prefix argument付きの =C-u C-c RET= は
Hermesを有効化してからgptel標準の送信menuを開きます。

** ユーザーが使う主なコマンド

| コマンド | 用途 |
|----------+---------------------------------------------|
| =gptel-hermes-send= | 現在のbufferでHermesを有効化して送信 |
| =gptel-hermes-global-send-mode= | 全bufferの =C-c RET= をHermes送信にする |
| =gptel-hermes-enable= | 現在のbufferにHermes promptとtoolを設定 |
| =gptel-hermes-workspace-root= | workspaceのパスを表示 |
| =gptel-hermes-set-workspace= | workspaceを明示設定 |
| =gptel-hermes-workspace-status= | workspace設定状態を表示 |
| =gptel-hermes-clear-workspace= | workspaceを解除 |
| =gptel-hermes-prompt-inspect= | 実際のsystem promptを別バッファで確認 |
| =gptel-hermes-migrate-skill-overlay= | 旧形式のskillコピーを安全に移行 |
| =gptel-hermes-skill-customize= | bundled skillをuser overlayへcopy-on-write |
| =gptel-hermes-skill-diff-bundled= | user版とbundled版の差分を確認 |

* 設定

** スキルディレクトリ

デフォルトは =~/.gptel-hermes/profiles/main/skills= です。既存のHermes Agentが使用する =~/.hermes/= 以下のスキルツリーとは分離して管理します。

通常は設定不要ですが、初期設定を明示する場合は次のように指定します。

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/.gptel-hermes/profiles/main/skills")
#+end_src

パッケージ内の bundled =skills/= を直接使う場合（ローカル開発時など）:

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/YOUR-CODE-PATH/gptel-hermes/skills")
#+end_src

別の既存Hermes Agentスキルツリーを使う場合:

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/.hermes/profiles/main/skills")
#+end_src

パッケージ外のディレクトリ（デフォルトの =~/.gptel-hermes/profiles/main/skills= を含む）はユーザーoverlayとして扱われます。=gptel-hermes-enable= はコピーを行いません。各 =SKILL.md= はユーザーoverlayを優先し、存在しなければパッケージ内の bundled =skills/= を使います。参照リソースも同じ相対パス単位でユーザー版からbundled版へフォールバックします。

このリポジトリの bundled =skills/= はread-onlyの基準です。=hermes_skill_update= は完全な =SKILL.md= を検証してからoverlayへcopy-on-writeし、bundledのSHA-256を =.gptel-hermes-origin= に記録します。bundled更新は自動mergeせず、=hermes_skill_view= でupstream変更を報告します。

既存の旧形式コピーを移行する場合:

#+begin_src emacs-lisp
M-x gptel-hermes-migrate-skill-overlay
#+end_src

旧markerを検出すると =gptel-hermes-enable= が警告します。移行コマンドは確認後、bundledと同一の旧コピーを削除し、差分のある旧コピーをoverlay外のbackupへ保存してから削除します。bundledにないユーザー作成Skillは保持します。完全なskillディレクトリを明示的にユーザーoverlayへコピーする場合は =gptel-hermes-skill-customize= を使います。bundledとの差分は =gptel-hermes-skill-diff-bundled= で確認できます。

旧コマンド =gptel-hermes-reinstall-skills= は互換性のため残していますが、通常のoverlay運用では使いません。

#+begin_src emacs-lisp
M-x gptel-hermes-reinstall-skills
#+end_src

実行前に確認を求め、bundledと同じ相対パスの =SKILL.md= だけを上書きします。bundledに存在しないユーザー独自Skillは削除しません。

** Skillの作成と検証

- =hermes_skill_validate= は、指定した相対Skill IDの =SKILL.md= を読み取り専用で検証します。name、description、frontmatterの区切り、本文、サイズ制限などの違反を一覧で返します。確認は不要です。
- =hermes_skill_resource_path= は、bundledまたはuser overlayのresourceを実効絶対パスへ解決します。結果に表示されたSkill要求terminalへ、Skill提供scriptを渡してください。
- =hermes_skill_create= は、Skill ID、description、本文から最小限のfrontmatterを生成し、検証してから =gptel-hermes-skills-directory/<skill-id>/SKILL.md= に新規作成します。既存ファイルを上書きせず、作成先は =gptel-hermes-skills-directory= 配下に限られ、確認が必要です。

=gptel-hermes-enable= を実行すると、Skillを検証し、system promptのSkill索引を再生成します。全文本文は =hermes_skill_view= で必要時に読み込みます。個別の違反内容を確認する場合は =hermes_skill_validate= を使ってください。

bundled Skillのfrontmatterには、利用するmodel-facing toolを =requires_tools: [tool-a, tool-b]= として列挙します。=gptel-hermes-enable= は、Hermes標準toolと既存のbuffer-local =gptel-tools= を統合したeffective tool listで依存性を判定し、missing toolがあるSkillを標準の索引から除外します。ユーザーが同名のtoolを追加している場合は、そのtoolで依存性を満たせます。除外されたSkillも =hermes_skill_view= を明示すれば本文とrequired/missing tool一覧を確認できます。

browser操作、web search、session search、委譲、PTY、background processはこのpackageの標準toolではありません。これらを必須とするbundled Skillは =requires_tools= に外部tool名を持つため、対応toolのないbufferでは索引に出ません。user overlayの =SKILL.md= は自動変換せず、=requires_tools= を省略した場合は依存なしとして後方互換で扱います。

runtimeのtoolは =hermes_file_read=、=hermes_file_write=、=hermes_apply_patch=、=hermes_terminal=、=hermes_elisp_call= です。Skill overlay更新の =hermes_skill_update= も確認付きで提供します。読み取り専用のElisp callは固定allowlistだけを受け付けます。認証済みCLI用の =hermes_terminal_authenticated= はデフォルトではtool登録されず、実HOMEとEmacsの環境変数を継承するため、明示的に次を設定した場合だけ確認付きで登録されます。

#+begin_src emacs-lisp
(setq gptel-hermes-enable-authenticated-terminal t)
#+end_src

このtoolを使うSkillは =requires_tools= に同toolを宣言します。通常の =hermes_terminal= はtemporary HOMEと限定環境を維持するため、認証情報を必要としないCLI用です。任意Elisp評価の =hermes_elisp_eval= もデフォルトではtool登録されず、明示的に次を設定した場合だけ確認付きで登録されます。

#+begin_src emacs-lisp
(setq gptel-hermes-enable-unsafe-elisp-eval t)
#+end_src

任意Elisp評価はEmacsプロセスの全権限を持ちます。 =gptel-confirm-tool-calls= を =nil= にするとgptel側で確認を無効化できますが、これは意図的なユーザー設定です。

bundled / リポジトリの =skills/= はリポジトリのsource領域です。gptel-hermesのSkill作成toolから直接作成・上書きする対象ではありません。そこへSkillを追加・変更する場合は、明示的なリポジトリ編集として通常の編集、テスト、commitを行ってください。

** メモリディレクトリ

=MEMORY.md= と =USER.md= は、デフォルトでは既存のHermes Agentと分離した以下から読み書きします。

#+begin_example
~/.gptel-hermes/memories/
#+end_example

環境変数 =HERMES_HOME= は暗黙には参照しません。既存のHermes Agentのメモリを明示的に使いたい場合だけ、 =gptel-hermes-home= を設定してください。

#+begin_src emacs-lisp
(setq gptel-hermes-home "~/.hermes/profiles/main")
#+end_src

* 内部動作

** =gptel-hermes-enable= が裏で行うこと

有効化は単にpromptへ文字列を追加するだけではありません。現在のbufferで次の処理を行います。

1. 初回だけ、初期 =default-directory= からworkspaceを決めてbuffer-localに保存する。以後、terminal実行などで =default-directory= が変わってもworkspaceは変えない。
2. ユーザーoverlayとbundled =skills/= を相対Skill IDごとに調べ、ユーザー版を優先して索引を再生成する。ユーザー側にないSkillやresourceはbundled側へフォールバックする。
3. すべての =SKILL.md= のfrontmatterとサイズを検証し、違反があれば有効化を止めずに件数をmessageへ出す。詳細は =hermes_skill_validate= で確認する。
4. effective tool listを確定し、=requires_tools= のmissing toolがあるSkillを索引から除外する。messageにはindexed、incompatible、invalidの件数を出す。
5. =MEMORY.md=、=USER.md=、compatibleなSkillの名前・description・categoryだけをsystem promptへ組み立てる。全文 =SKILL.md= はpromptへ入れない。
6. 最初に有効化した時点の元system promptをbuffer-localに保存し、以後の再有効化で重複追加しない。
7. このバッファの =gptel-tools= にHermes toolを追加する。同名toolは重複登録せず、unsafeな =hermes_elisp_eval= は明示設定が有効なときだけ作成する。
8. 旧形式の =.gptel-hermes-bundled-skills= markerがある場合は、bundled更新をshadowする可能性をwarningで知らせる。

** runtime toolの共通経路

runtime toolはPython daemonや別のagent processではなく、Emacs Lispからgptelのtool callbackとして実行されます。戻り値はgptelが扱える境界付きのプレーンテキストです。

- Fileとpatchは共通のpath resolverを通る。 =..= 、workspace外、TRAMP、symlink経由の脱出、 =.git/= や秘密ファイルを拒否し、canonical pathでも再度secret判定する。
- =hermes_file_read= はregular fileだけを読み、8 MiBを超える入力、NULを含むbinary、範囲外の行指定を拒否する。結果はSHA-256、行範囲、128 KiBでのtruncate状態を含む。
- =hermes_file_write= は1 MiBまでで、同じdirectoryにtemporary fileを書いてrenameするatomic writeを使う。=replace= は読み取り時のSHA-256を要求し、stale writeを拒否する。
- =hermes_apply_patch= はGit repositoryのtop-levelだけを対象にする。patchのrename/copyを含む全pathを検証し、=git apply --check= を通してから、check-onlyでなければ確認後に適用する。check-onlyではファイルを変更しない。
- =hermes_terminal= はshell文字列ではなくprogramとargv配列を =make-process= へ渡す。temporary HOME、限定した環境変数、最大300秒のtimeout、stdout/stderr各64 KiBの上限を使い、stdinは閉じる。OS sandboxではない。
- =hermes_terminal_authenticated= は明示的に有効化した場合だけ登録され、実HOMEとEmacsの環境変数を継承する。認証済みCLIを実行できる代わりに、呼び出したprogramからローカル認証情報へ到達できるため、確認付きで実行する。
- =hermes_elisp_call= は文字列のallowlistにある読み取り専用関数だけを呼ぶ。 =hermes_elisp_eval= はデフォルトではtool登録されず、有効化するとEmacsプロセスの全権限でexactly one formを評価する。

既定の確認ポリシーは、読み取り専用のskill/file-read/agenda/allowlisted-Elisp callは確認なし、ファイル書き込み・patch適用・terminal・memory・Org変更・skill作成/更新は確認ありです。 =gptel-confirm-tool-calls= を =nil= にするとgptel側で全確認を無効化できます。

** 初期 prompt

有効化時、以下を system prompt に snapshot として注入します。

- スキル名
- description
- category
- =MEMORY.md=
- =USER.md=

=SKILL.md= 本文は最初の prompt には入れません。

** スキル本文

モデルが必要なスキルを選び、 =hermes_skill_view= を呼びます。

#+begin_example
hermes_skill_view(name="software-development/my-ponytail")

hermes_skill_view(name="software-development/my-ponytail", resource="references/example.txt")
#+end_example

その結果として、対象 =SKILL.md= または相対resourceの本文が tool result としてモデルへ返ります。resourceはユーザーoverlayを優先し、なければbundled側へフォールバックします。
=hermes_skill_resource_path= の結果には、実際に選択されたoverlayまたはbundled resourceの =Skill directory= と =Effective path= が含まれます。resource fileの親からrootを推測せず、shell例では =Skill directory= を使い、個別実行では =Effective path= をterminalの引数に渡します。

** メモリ

=hermes_memory= は次の操作を提供します。

- =add=
- =replace=
- =remove=

対象は以下です。

- =memory= → =MEMORY.md=
- =user= → =USER.md=

更新は検証後に atomic write されます。

* Orgタスク管理

Orgファイルを正本としたタスク管理を、専用のgptel toolで利用できます。
=org-agenda-files= は現在のEmacs設定を正本とし、tool呼び出しごとに再評価します。設定されていない場合、ホームディレクトリや =~/org= を自動探索しません。

** Orgファイルの設定

少なくとも一つの既存ファイルを =org-agenda-files= に設定してください。

#+begin_src emacs-lisp
(setq org-agenda-files
      (list "~/org/work.org"
            "~/org/home.org"))
#+end_src

必要な場合だけ、 =org-agenda-files= が空のときに =org-directory= をフォールバックとして使えます。

#+begin_src emacs-lisp
(setq gptel-hermes-org-directory-fallback t)
#+end_src

この設定も =org-directory= 以下を無条件に再帰探索するものではなく、Orgのagendaファイル規則に従います。書き込み対象は解決済みのagendaファイルに限られます。

** Capture template

captureを使う場合は、既存の =org-capture-templates= を設定してください。保存先をSKILLやtool呼び出し側で直接指定するのではなく、templateの設定を使います。

#+begin_src emacs-lisp
(setq org-capture-templates
      '(("t" "Task" entry
         (file+headline "work.org" "Tasks")
         "* TODO %?")))
#+end_src

相対パスは =org-directory= 基準です。templateの保存先も =org-agenda-files= に含めてください。template keyが存在しない場合は、toolが利用可能なkeyを返します。capture template自体が未設定の場合は、設定方法を案内して終了します。

** Toolの利用

まず読み取り専用の =hermes_org_agenda= を呼び出します。

#+begin_example
hermes_org_agenda(view="open", limit=100)
hermes_org_agenda(view="tag", tag="202607")
#+end_example

=open= は未完了、 =all= は全状態、 =tag= は指定タグのタスクを返します。返却値にはファイル、行番号、見出し、現在のTODO状態、タグ、優先度、期限・予定情報が含まれます。 =YYYYMM= タグを付けておくと月次レビューに利用できます。

状態変更とcaptureは確認付きの =hermes_org_task= を使います。

#+begin_example
hermes_org_task(action="todo",
                file="/Users/me/org/work.org",
                line=42,
                heading="API仕様を確認",
                keyword="DOIN")

hermes_org_task(action="capture",
                text="新しいタスクの内容",
                template="t")
#+end_example

状態変更では、直前の =hermes_org_agenda= のfile・line・完全一致するheadingと、現在のEmacs設定にあるkeywordを渡します。staleな行や見出し、agenda外ファイル、不正なkeywordは拒否されます。一回の呼び出しで一タスクだけを変更し、 =org-todo= と既存のログ・hookを利用します。一般のterminal/file toolでOrgファイルを書き換えず、必ずこの専用toolを使ってください。

この機能はバックグラウンド監視、通知、cron/launchd、Google Calendar・Slack・Backlog等との同期を提供しません。カレンダーに既に登録された予定をOrgへ二重登録しないでください。

* prompt の確認

現在の gptel バッファに設定された system prompt を確認します。

#+begin_src emacs-lisp
M-x gptel-hermes-prompt-inspect
#+end_src

現在の検証対象:

- 初期 prompt に索引・メモリ・プロフィールが入る
- 初期 prompt に全文 =SKILL.md= が入らない
- 選択した =SKILL.md= が tool result になる
- メモリの add / replace / remove が永続化される
- gptel バッファ内の prompt snapshot が安定する
- bundled/user overlayの優先順位とcopy-on-write更新
- bundled origin SHA-256の記録とupstream変更の報告
- workspace path containment、secret path、symlink escape
- file read/writeのサイズ制限、atomic write、stale SHA拒否
- Git unified patchのcheck-onlyと適用
- terminalのtimeout、環境サニタイズ、出力上限
- bundled Skillの =requires_tools=、compatible index filtering、legacy tool call監査
- allowlisted Elisp callと明示有効化されたunsandboxed eval
- bundled =SKILL.md= 全件がvalidationを通ること
- bundled =SKILL.md= の確認付き再インストールとユーザー独自Skillの保持
