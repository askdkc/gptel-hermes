#+title: gptel-hermes
#+author: dkc

* 概要

=gptel-hermes= は、Nous Researchが開発したHermes Agentの機能の一部を、Emacsのgptel経由のチャットで利用可能にする独自のEmacs Lispパッケージです。

#+begin_example
Emacs / gptel
  └── gptel-hermes.el
       ├── skills/**/SKILL.md の索引
       ├── system prompt への索引・メモリ注入
       ├── hermes_skill_view tool
       ├── hermes_skill_validate tool
       ├── hermes_skill_create tool
       ├── hermes_memory tool
       ├── hermes_org_agenda tool
       └── hermes_org_task tool
#+end_example

Hermes Agentの機能をEmacsで利用するため、gptel向けに独自実装しています。

* セットアップ

** use-package と :vc でインストール（Emacs 30以降）

Emacs 30以降では、組み込みの =use-package= と =:vc= を使ってGitHubからインストールできます。初回評価時にリポジトリが取得され、=gptel-hermes= がロードされます。

#+begin_src emacs-lisp
(use-package gptel-hermes
  :vc (:url "https://github.com/askdkc/gptel-hermes.git"
       :rev :newest)
  :custom
  (gptel-hermes-skills-directory
   (expand-file-name "~/.gptel-hermes/profiles/main/skills"))
  (gptel-hermes-home
   (expand-file-name "~/.gptel-hermes"))
  :config
  (add-hook 'gptel-mode-hook #'gptel-hermes-enable))
#+end_src

この方法にはGitが必要です。=gptel-hermes-skills-directory= と =gptel-hermes-home= は既存のHermes Agentの環境と分離した上記のディレクトリを初期設定として推奨します。設定しない場合も同じディレクトリがデフォルトで使われます。

** 手動でインストール

#+begin_src emacs-lisp
(add-to-list 'load-path "~/YOUR-CODE-PATH/gptel-hermes")
(require 'gptel-hermes)
(add-hook 'gptel-mode-hook #'gptel-hermes-enable)
#+end_src

gptel バッファで手動有効化する場合:

#+begin_src emacs-lisp
M-x gptel-hermes-enable
#+end_src

* 設定

** スキルディレクトリ

デフォルトは =~/.gptel-hermes/profiles/main/skills= です。既存のHermes Agentが使用する =~/.hermes/= 以下のスキルツリーとは分離して管理します。

通常は設定不要ですが、初期設定を明示する場合は次のように指定します。

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/.gptel-hermes/profiles/main/skills")
#+end_src

パッケージ内の bundled =skills/= を直接使う場合（ローカル開発時など）:

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/YOUR-CODE-PATH/gptel-hermes/skills")
#+end_src

別の既存Hermes Agentスキルツリーを使う場合:

#+begin_src emacs-lisp
(setq gptel-hermes-skills-directory
      "~/.hermes/profiles/main/skills")
#+end_src

パッケージ外のディレクトリ（デフォルトの =~/.gptel-hermes/profiles/main/skills= を含む）を指定した場合、最初の =gptel-hermes-enable= 時に、パッケージ内の bundled =skills/= にある索引対象の =SKILL.md= だけが、同じ相対パスで指定先へコピーされます。指定先や親ディレクトリが存在しなくても自動作成されます。指定先に同名のファイルが既にある場合は上書きせず、そのファイルを優先します。コピーが完了すると指定先に =.gptel-hermes-bundled-skills= マーカーが作成され、以後の有効化やパッケージ更新で自動的に上書き・再同期されることはありません。

コピー対象は現在のスキル索引と同じ =SKILL.md= のみです。=DESCRIPTION.md=、=references/=、=scripts/= などはコピーされません。同期結果（コピー数、既存ファイル保持数、同期済みでスキップしたこと）はミニバッファの =message= で通知されます。

このリポジトリの bundled =skills/= には基本スキルセットを配置済みです。bundled =skills/= 自体を指定した場合は、自己コピーを行いません。

** Skillの作成と検証

- =hermes_skill_validate= は、指定した相対Skill IDの =SKILL.md= を読み取り専用で検証します。name、description、frontmatterの区切り、本文、サイズ制限などの違反を一覧で返します。確認は不要です。
- =hermes_skill_create= は、Skill ID、description、本文から最小限のfrontmatterを生成し、検証してから =gptel-hermes-skills-directory/<skill-id>/SKILL.md= に新規作成します。既存ファイルを上書きせず、作成先は =gptel-hermes-skills-directory= 配下に限られ、確認が必要です。

=gptel-hermes-enable= を実行すると、Skillを検証し、system promptのSkill索引を再生成します。全文本文は =hermes_skill_view= で必要時に読み込みます。個別の違反内容を確認する場合は =hermes_skill_validate= を使ってください。

bundled / リポジトリの =skills/= はリポジトリのsource領域です。gptel-hermesのSkill作成toolから直接作成・上書きする対象ではありません。そこへSkillを追加・変更する場合は、明示的なリポジトリ編集として通常の編集、テスト、commitを行ってください。

** メモリディレクトリ

=MEMORY.md= と =USER.md= は、デフォルトでは既存のHermes Agentと分離した以下から読み書きします。

#+begin_example
~/.gptel-hermes/memories/
#+end_example

環境変数 =HERMES_HOME= は暗黙には参照しません。既存のHermes Agentのメモリを明示的に使いたい場合だけ、=gptel-hermes-home= を設定してください。

#+begin_src emacs-lisp
(setq gptel-hermes-home "~/.hermes/profiles/main")
#+end_src

* 動作

** 初期 prompt

有効化時、以下を system prompt に snapshot として注入します。

- スキル名
- description
- category
- =MEMORY.md=
- =USER.md=

=SKILL.md= 本文は最初の prompt には入れません。

** スキル本文

モデルが必要なスキルを選び、=hermes_skill_view= を呼びます。

#+begin_example
hermes_skill_view("software-development/my-ponytail")
#+end_example

その結果として、対象 =SKILL.md= の本文が tool result としてモデルへ返ります。

** メモリ

=hermes_memory= は次の操作を提供します。

- =add=
- =replace=
- =remove=

対象は以下です。

- =memory= → =MEMORY.md=
- =user= → =USER.md=

更新は検証後に atomic write されます。

* Orgタスク管理

Orgファイルを正本としたタスク管理を、専用のgptel toolで利用できます。
=org-agenda-files= は現在のEmacs設定を正本とし、tool呼び出しごとに再評価します。設定されていない場合、ホームディレクトリや =~/org= を自動探索しません。

** Orgファイルの設定

少なくとも一つの既存ファイルを =org-agenda-files= に設定してください。

#+begin_src emacs-lisp
(setq org-agenda-files
      (list "~/org/work.org"
            "~/org/home.org"))
#+end_src

必要な場合だけ、=org-agenda-files= が空のときに =org-directory= をフォールバックとして使えます。

#+begin_src emacs-lisp
(setq gptel-hermes-org-directory-fallback t)
#+end_src

この設定も =org-directory= 以下を無条件に再帰探索するものではなく、Orgのagendaファイル規則に従います。書き込み対象は解決済みのagendaファイルに限られます。

** Capture template

captureを使う場合は、既存の =org-capture-templates= を設定してください。保存先をSKILLやtool呼び出し側で直接指定するのではなく、templateの設定を使います。

#+begin_src emacs-lisp
(setq org-capture-templates
      '(("t" "Task" entry
         (file+headline "work.org" "Tasks")
         "* TODO %?")))
#+end_src

相対パスは =org-directory= 基準です。templateの保存先も =org-agenda-files= に含めてください。template keyが存在しない場合は、toolが利用可能なkeyを返します。capture template自体が未設定の場合は、設定方法を案内して終了します。

** Toolの利用

まず読み取り専用の =hermes_org_agenda= を呼び出します。

#+begin_example
hermes_org_agenda(view="open", limit=100)
hermes_org_agenda(view="tag", tag="202607")
#+end_example

=open= は未完了、=all= は全状態、=tag= は指定タグのタスクを返します。返却値にはファイル、行番号、見出し、現在のTODO状態、タグ、優先度、期限・予定情報が含まれます。=YYYYMM= タグを付けておくと月次レビューに利用できます。

状態変更とcaptureは確認付きの =hermes_org_task= を使います。

#+begin_example
hermes_org_task(action="todo",
                file="/Users/me/org/work.org",
                line=42,
                heading="API仕様を確認",
                keyword="DOIN")

hermes_org_task(action="capture",
                text="新しいタスクの内容",
                template="t")
#+end_example

状態変更では、直前の =hermes_org_agenda= のfile・line・完全一致するheadingと、現在のEmacs設定にあるkeywordを渡します。staleな行や見出し、agenda外ファイル、不正なkeywordは拒否されます。一回の呼び出しで一タスクだけを変更し、=org-todo= と既存のログ・hookを利用します。一般のterminal/file toolでOrgファイルを書き換えず、必ずこの専用toolを使ってください。

この機能はバックグラウンド監視、通知、cron/launchd、Google Calendar・Slack・Backlog等との同期を提供しません。カレンダーに既に登録された予定をOrgへ二重登録しないでください。

* prompt の確認

現在の gptel バッファに設定された system prompt を確認します。

#+begin_src emacs-lisp
M-x gptel-hermes-prompt-inspect
#+end_src

スキル索引とメモリは確認できますが、全文の =SKILL.md= はモデルが =hermes_skill_view= を呼ぶまで入りません。

* 検証

gptel の checkout を load-path に指定して実行します。

#+begin_src sh
emacs --batch -L /path/to/gptel -L . -f batch-byte-compile gptel-hermes.el
emacs --batch -L /path/to/gptel -L . -L tests \
  -l tests/gptel-hermes-tests.el \
  -f ert-run-tests-batch-and-exit
#+end_src

- macOSの場合
  macOSの場合はemacsコマンドは =/Applications= ディレクトリ配下になります。
  #+begin_src sh
  /Applications/Emacs.app/Contents/MacOS/Emacs --batch -L /path/to/gptel -L . -f batch-byte-compile gptel-hermes.el
  /Applications/Emacs.app/Contents/MacOS/Emacs --batch -L /path/to/gptel -L . -L tests \
  -l tests/gptel-hermes-tests.el \
  -f ert-run-tests-batch-and-exit
  #+end_src

現在の検証対象:

- 初期 prompt に索引・メモリ・プロフィールが入る
- 初期 prompt に全文 =SKILL.md= が入らない
- 選択した =SKILL.md= が tool result になる
- メモリの add / replace / remove が永続化される
- gptel バッファ内の prompt snapshot が安定する
- 外部スキルディレクトリへの初回 bundled =SKILL.md= 同期
- 既存 =SKILL.md= の保持と同期完了マーカー
- 同期済みディレクトリでの再コピー抑止
- 同期途中のエラー時に完了マーカーを作成しない
- bundled =skills/= 自体を指定した場合の自己コピー回避

* 現在の未実装機能

Hermes Agentと違う未実装な部分:

- =session_search=
- =delegate_task=
- terminal / file 編集 tool
- background review
- 外部 memory provider

これらは gptel の skills / memory 経路とは別の agent runtime 機能です。
